# /brain — 브레인스토밍 워크플로우

사용자가 `/brain <주제>`를 실행하면 아래 3단계를 순서대로 진행합니다.
코드를 작성하지 않고, 의사결정만 수행합니다.

---

## Phase 1: 브레인스토밍 (발산→수렴)

### Step 0) 문제 정의
- 사용자의 `<주제>`를 1문장으로 재정의하고 확인한다.
- 성공 기준("Done")을 한 줄로 정의한다.

### Step 1) 제약/맥락 수집 (최소 질문)
아래 중 비어 있는 것을 우선으로 **1~3개만** 질문 (AskUserQuestion 사용):
- 사용자/고객은 누구? 사용 시나리오 1개만
- 비용/시간/리소스 제약?
- 반드시 지켜야 할 정책/보안/규정?
- 이미 있는 시스템/스택 제약?

### Step 2) 대안 생성 (발산)
- 대안 **2~4개**를 만든다. ("한 가지 답" 강요 금지)
- 각 대안마다: **핵심 아이디어 / 장점 / 단점 / 실패 모드(어떻게 망하나)**

### Step 3) 비교/선택 (수렴)
- 비교 기준 4~6개를 **표**로 만들어 점수/서술 평가
- 추천안을 1개로 좁히되, "언제 추천이 뒤집히는가(조건)"도 함께 명시

---

## Phase 2: Design Brief 출력 (반드시 이 포맷 준수)

```markdown
# Brainstorm 결과: <주제/기능명>

## 1) 목표(1문장) / 성공 기준(DoD)
- 목표:
- 성공 기준:

## 2) 전제/제약
- 전제:
- 제약:

## 3) 대안(2~4개)
### A안: <이름>
- 핵심:
- 장점:
- 단점:
- 실패 모드:
### B안: <이름>
...

## 4) 비교(기준 4~6개)
| 기준 | A안 | B안 | C안 |
|------|-----|-----|-----|
| 비용 | ... | ... | ... |
| 리드타임 | ... | ... | ... |
| 리스크 | ... | ... | ... |
| 확장성 | ... | ... | ... |
| 운영 난이도 | ... | ... | ... |
| 사용자 가치 | ... | ... | ... |

## 5) 추천안 + 이유(핵심 3줄)
- 추천:
- 이유:
- 뒤집히는 조건:

## 6) 리스크/완화(Top 3)
- R1 / 완화:
- R2 / 완화:
- R3 / 완화:

## 7) PDCA Handoff
- 다음 명령: `/pdca plan <feature-name>`
- plan 문서에 넣을 핵심 포인트 (bullet 5~10개)
- (선택) design 핵심 포인트 (bullet 5~10개)
```

---

## Phase 3: PDCA plan handoff

결과 출력 후 **반드시**:
1. feature name을 `kebab-case`로 제안
2. "`/pdca plan <feature-name>` 으로 계획 문서를 생성하시겠습니까?" 안내
3. plan 문서에 넣을 핵심 포인트 bullet 5~10개 정리

---

## 규칙
- 이 워크플로우에서는 **코드를 작성하지 않는다**. 의사결정만 한다.
- 질문은 한 번에 1~3개만 (사용자 피로 방지).
- 결론에는 반드시 "리스크 + 완화책"을 포함한다.
