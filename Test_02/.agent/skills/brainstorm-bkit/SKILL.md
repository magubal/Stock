# Brainstorming for bkit (발산→수렴→handoff)

## 언제 이 스킬을 쓰나
- "아이디어가 필요해", "대안을 비교해줘", "방향성 잡아줘", "뭐가 더 나을까?",같은 요청
- 요구가 모호해서 바로 /pdca plan을 쓰기 전에 '선택지와 기준'을 정해야 할 때
- 리서치/프로덕트/기술 의사결정에서 옵션 트레이드오프가 핵심일 때

## 기본 원칙
1) 질문은 한 번에 1~3개만(사용자 피로 방지)
2) 대안은 최소 2개, 보통 3개("한 가지 답" 강요 금지)
3) 비교 기준을 먼저 합의(속도/비용/리스크/확장성/운영 난이도 등)
4) 결론은 "추천 + 왜 + 리스크/완화"까지 포함
5) 코딩/구현 지시는 여기서 하지 말고, 결정이 나면 bkit PDCA로 넘긴다

## 진행 프로토콜
### Step 0) 문제 정의
- 사용자의 목표를 1문장으로 재정의하고 확인한다.
- 성공 기준("Done")을 한 줄로 정의한다.

### Step 1) 제약/맥락 수집(최소 질문)
아래 중 비어 있는 것을 우선으로 1~3개 질문:
- 사용자/고객은 누구? 사용 시나리오 1개만
- 비용/시간/리소스 제약?
- 반드시 지켜야 할 정책/보안/규정?
- 이미 있는 시스템/스택 제약?

### Step 2) 대안 생성(발산)
- 대안 2~4개를 만든다.
- 각 대안마다 "핵심 아이디어 / 장점 / 단점 / 실패 모드(어떻게 망하나)"를 적는다.

### Step 3) 비교/선택(수렴)
- 비교 기준 4~6개를 표로 만들어 점수/서술 평가
- 추천안을 1개로 좁히되,
  - "언제 추천이 뒤집히는가(조건)"도 함께 명시한다.

### Step 4) 산출물(Design Brief) 생성
아래 포맷으로 최종 정리:
